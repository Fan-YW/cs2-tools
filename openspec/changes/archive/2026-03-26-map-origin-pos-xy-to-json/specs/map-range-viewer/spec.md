## 新增需求

### 需求:左侧与上侧刻度与坐标标注

系统必须在左侧与上侧（地图展示框的左边界与上边界）绘制刻度与坐标数值标注。

系统必须使用当前地图元数据中的 `pos_x`、`pos_y` 作为“左上角实际世界坐标”。其中 `pos_x`、`pos_y` 分别对应地图可视区域的左边界与上边界在世界坐标系中的值，并用于刻度数值的换算。

系统必须根据当前缩放倍率（滚轮缩放后的等效“每屏幕像素对应的游戏单位数”，记为 `X`）动态确定刻度间距，刻度的选取规则必须满足以下约束：

1. **主刻度（Major）**
   - 主刻度的间距（世界单位步长）`majorStep` 必须取自集合 `1/2/5 * 10^n`（`n` 为整数，且该集合对应“主刻度取值因子”）。
   - 主刻度的间距应随 `X` 改变而变化，使得主刻度在屏幕上的间距大致落在可读范围（实现可通过选择使 `majorStep` 最接近 `targetMajorPx * X` 的候选值达成，`targetMajorPx` 为实现固定常量）。
2. **副刻度（Minor）**
   - 当 `majorStep` 的首位因子为 `1` 或 `5` 时，副刻度步长 `minorStep = majorStep / 5`，因此相邻两主刻度之间应出现 4 个副刻度。
   - 当 `majorStep` 的首位因子为 `2` 时，副刻度步长 `minorStep = majorStep / 4`，因此相邻两主刻度之间应出现 3 个副刻度。
3. **标注**
   - 必须在主刻度位置绘制坐标数值（世界单位），副刻度不标注数值或使用同等清晰策略（但必须与现有视觉层级一致）。
   - 当主刻度位置与世界坐标对应时，坐标数值必须与世界坐标一致（允许实现使用合理的取整策略）。

#### 场景:缩放/平移后刻度更新

- **当** 用户在地图展示区域滚轮缩放或左键拖动平移
- **那么** 左侧与上侧的主/副刻度间距与标注的数值必须同步更新，且刻度应始终与当前可视区域的世界坐标对齐

#### 场景:主刻度首位为 1/5 时的副刻度数量

- **当** 当前计算得到的 `majorStep` 的首位因子为 `5`（即 `majorStep = 5 * 10^n`）
- **那么** 相邻两主刻度之间应出现 4 个副刻度，且副刻度数值步长为 `majorStep/5`
- **并且** 当可视区域跨越主刻度 `3000、3500、4000` 时，副刻度必须包含 `3100、3200...`（具体包含范围受可视窗口截断影响）

#### 场景:主刻度首位为 2 时的副刻度数量

- **当** 当前计算得到的 `majorStep` 的首位因子为 `2`（即 `majorStep = 2 * 10^n`）
- **那么** 相邻两主刻度之间应出现 3 个副刻度，且副刻度数值步长为 `majorStep/4`

## 修改需求

### 需求:地图元数据为 JSON

系统必须从 **`public/map/json/`** 目录下与地图同名的 `.json` 文件加载元数据（例如 `de_dust2.json`）。该 JSON 必须由实现流程中的 **txt 转 JSON** 步骤从各源 `.txt` 生成或维护，并**归档**到 `public/map/json/`；**不得**在运行时以未结构化的 txt 作为唯一数据源。JSON 至少必须包含 `scale`、`c4_base_damage`、`pos_x`、`pos_y`（数值类型），且必须能唯一对应所选地图基名。其中 `pos_x`、`pos_y` 表示地图左上角世界坐标，供左侧/上侧刻度与坐标标注的换算使用。雷达 PNG 必须**归档**在 **`public/map/png/`** 目录（与 `public/map/json` 并列，均在 `public/map` 下），页面不得依赖将 PNG 与 JSON 混放在未区分的单一目录中。

#### 场景:加载 JSON 元数据

- **当** 用户选择某地图且存在对应的 `.json`
- **那么** 系统必须解析 JSON 并获得 `scale` 与 `c4_base_damage` 用于倍率与伤害计算，同时获得 `pos_x` 与 `pos_y` 用于刻度与坐标换算
