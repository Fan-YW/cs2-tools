## 上下文

`map-range-viewer` 的左侧/上侧刻度与坐标标注需要地图左上角世界坐标 `pos_x`/`pos_y`，并将其与当前缩放倍率结合来换算可视范围内的主/副刻度数值与位置。

当前实现会在运行时额外从 `public/map/raw/<map>.txt` 解析 `pos_x`/`pos_y`，而项目规范要求页面使用 `public/map/json/<map>.json` 作为结构化元数据来源（避免以未结构化 txt 作为唯一数据源）。

本变更将 `pos_x`/`pos_y` 纳入 `txt -> json` 产物，并调整前端读取路径。

## 目标 / 非目标

**目标：**
- `public/map/json/<map>.json` 中包含 `pos_x`、`pos_y`（数值类型，表示左上角世界坐标）。
- 前端用于刻度/坐标换算的 `pos_x/pos_y` 来自已加载的 `public/map/json/<map>.json`，不再依赖运行时解析源 txt。
- 保持主/副刻度的计算规则随放大倍率变化（主刻度取 `1/2/5 * 10^n`；主刻度因子为 `1/5` 时每段有 4 个副刻度、为 `2` 时每段有 3 个副刻度）。

**非目标：**
- 不重写刻度绘制的可视逻辑（仅更换数据来源：由 json 提供 `pos_x/pos_y`）。
- 不调整现有地图 PNG/布局/缩放/测距/伤害计算等功能。

## 决策

1. 扩展 `scripts/map-txt-to-json.mjs` 的解析与输出
   - 决策：从 `public/map/raw/<map>.txt` 解析 `pos_x`、`pos_y`，并写入 `public/map/json/<map>.json`。
   - 理由：让“结构化元数据只来自 json”的约束在数据层面成立，同时能复用现有转换流程。
   - 替代方案：只在前端继续解析 txt。该方案会保留运行时对未结构化数据的依赖，不满足规范化目标。

2. 修改 `js/map-range.js` 的加载逻辑（`pages/map-range.html` 引入）
   - 决策：页面从 `public/map/json/<map>.json` 读取 `scale`、`c4_base_damage` 以及新增的 `pos_x`、`pos_y`，并把后两者作为刻度/坐标换算的输入。
   - 理由：刻度/坐标渲染是当前最依赖 pos_x/pos_y 的模块，使用 json 能减少运行时请求与解析，并与规范保持一致。
   - 替代方案：保留 `loadMapOriginFromTxt` 作为回退。可以作为兼容手段，但主路径仍以 json 为准（若 json 不含 pos 字段则刻度降级）。

3. 刻度主/副间距规则保持现有实现逻辑
   - 决策：保留现有 `drawWorldTicks()` 的“主刻度取 `1/2/5 * 10^n` 且根据主刻度因子决定副刻度数量”的实现方式。
   - 理由：当前实现已经符合你给出的副刻度数量约束（1/5 因子 -> `major/5`，2 因子 -> `major/4`，从而分别产生 4 或 3 个副刻度）。
   - 替代方案：重新实现刻度因子计算。没有必要，且可能引入回归。

## 风险 / 权衡

- 风险：已有 `public/map/json/*.json` 文件当前不包含 `pos_x/pos_y`。
  - 缓解：更新任务中要求运行转换脚本并生成更新后的 json，确保生产页面有完整字段。
- 风险：txt 中 `pos_x/pos_y` 的格式可能包含小数位/负号/额外空白，导致正则解析失败。
  - 缓解：在转换脚本中使用覆盖负号与可选小数的正则，并在缺失字段时记录明确错误/跳过。
- 风险：若前端遇到旧 json（无 pos 字段），刻度/坐标可能不显示。
  - 缓解：页面应当对 `pos_x/pos_y` 的缺失进行降级（不绘制刻度或显示明确的元数据错误）。

## Migration Plan

1. 运行 `node scripts/map-txt-to-json.mjs`，将 `pos_x/pos_y` 写入 `public/map/json/<id>.json`。
2. 更新 `js/map-range.js`：刻度/坐标换算直接使用加载到的 json 字段（`pos_x/pos_y`）。
3. 回归验证：
   - 切换不同地图后，左侧/上侧刻度与坐标可见且随缩放变化更新。
   - 主/副刻度数量符合主刻度因子规则（`1/5` -> 4 个副刻度，`2` -> 3 个副刻度）。
