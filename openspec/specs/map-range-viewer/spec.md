## 需求
### 需求:页面布局与地图视口尺寸

系统必须使用 **Vue 3** 实现页面结构，并采用 **Reka UI** 作为可复用交互组件与可访问性 primitives 的主要来源（允许在 Reka UI 之上使用项目级样式层统一视觉）。页面必须分为左右两栏：左侧区域必须用于地图展示，其高度必须基本占满浏览器可视区域高度，宽度必须等于其高度，从而呈现正方形视口。在桌面宽屏布局下，右侧信息栏（除左侧正方形地图以外的区域）必须在**水平方向**占满剩余可用空间；**不得**因对信息栏设置过窄的最大宽度而导致主内容区在信息栏右侧出现大面积仅背景或空白的区域（允许与页面根容器一致的极小内边距）。

#### 场景:视口为正方形且占满高度

- **当** 用户在桌面浏览器中打开页面且视口高度足够
- **那么** 左侧地图展示区域必须表现为正方形，且高度必须接近 100vh（允许为滚动条或边距留出极小余量）

#### 场景:宽屏下右栏占满剩余宽度

- **当** 用户在宽屏桌面浏览器中打开页面且视口宽度足以并排显示左侧正方形地图与右侧信息栏
- **那么** 右侧信息栏必须在水平方向上占满除左侧地图列以外的**全部**剩余可用空间，且不得在信息栏与视口右缘之间因仅限制信息栏最大宽度而留下大块空白

### 需求:地图选择与图像加载

系统必须在右侧栏最上方提供下拉选择框，选项必须对应下列七个地图基名（与源 Valve 概览 `.txt` 基名一致：`de_ancient`、`de_anubis`、`de_dust2`、`de_inferno`、`de_mirage`、`de_nuke`、`de_overpass`）。用户必须能够选择其中一张地图；选择后左侧必须从 **`public/map/png/`** 目录加载与该地图对应的 PNG 雷达图（与现有资源命名一致，例如 `de_dust2_radar_psd.png`）。**运行时 URL** 必须以**部署站点根**（Vite 构建产物根）为基准，通过 **`import.meta.env.BASE_URL`**（或项目内与之等价的封装）拼接 `public` 下文件的公共路径，使在 **GitHub Pages 子路径**（非根 `base`）部署时仍能正确解析到 `map/png/` 下资源。图像原始分辨率必须为 1024×1024 像素。地图图层必须**始终铺满**左侧正方形展示框；**禁止**在展示框内出现因未覆盖而产生的**白边**或底层背景露空（允许通过裁切图像边缘实现铺满；平移与缩放后仍须满足本条）。

#### 场景:切换地图更新图像

- **当** 用户从下拉框选择另一地图名称
- **那么** 左侧必须加载并显示该地图对应的 PNG，且必须重置或按规范更新缩放与测距状态（见清除与固定需求）

### 需求:地图元数据为 JSON

系统必须从 **`public/map/json/`** 目录下与地图同名的 `.json` 文件加载元数据（例如 `de_dust2.json`）。**运行时 URL** 必须以部署站点根为基准，通过 **`import.meta.env.BASE_URL`**（或等价封装）正确解析到 `map/json/` 下文件，以支持 GitHub Pages 子路径部署。该 JSON 必须由实现流程中的 **txt 转 JSON** 步骤从各源 `.txt` 生成或维护，并**归档**到 `public/map/json/`；**不得**在运行时以未结构化的 txt 作为唯一数据源。Valve 概览源 `.txt` 应保留在 **`public/map/raw/`**（与 `json`、`png` 并列，均在 `public/map` 下）；**允许**与同图雷达 PNG 并存，维护流程**不得**因已存在 PNG 而删除原始 txt。**txt 转 JSON 时**必须保留源文件中**全部** `"键" "值"` 对：实现必须在 JSON 中提供有序列表 **`entries`**（元素为 `[键字符串, 值字符串]`），顺序与源文件一致，**不得丢弃**任何键值对（含重复键的多次出现）。页面运行仍须能从该 JSON 读取数值字段 `scale`、`c4_base_damage`，以及用于刻度原点的 `pos_x`、`pos_y`（可与 `entries` 中最后一次同名键一致）。雷达 PNG 必须**归档**在 **`public/map/png/`** 目录，页面不得依赖将 PNG 与 JSON 混放在未区分的单一目录中。

#### 场景:加载 JSON 元数据

- **当** 用户选择某地图且存在对应的 `.json`
- **那么** 系统必须解析 JSON 并获得 `scale` 与 `c4_base_damage` 用于倍率与伤害计算，同时获得 `pos_x` 与 `pos_y`（若存在且为有效数值）用于刻度与坐标换算

### 需求:左侧与上侧刻度与坐标标注

系统必须在左侧与上侧（地图展示框的左边界与上边界）绘制刻度与坐标数值标注。

系统必须使用当前地图元数据中的 `pos_x`、`pos_y` 作为“左上角实际世界坐标”。其中 `pos_x`、`pos_y` 分别对应地图可视区域的左边界与上边界在世界坐标系中的值，并用于刻度数值的换算。

系统必须根据当前缩放倍率（滚轮缩放后的等效“每屏幕像素对应的游戏单位数”，记为 `X`）动态确定刻度间距，刻度的选取规则必须满足以下约束：

1. **主刻度（Major）**
   - 主刻度的间距（世界单位步长）`majorStep` 必须取自集合 `1/2/5 * 10^n`（`n` 为整数，且该集合对应“主刻度取值因子”）。
   - 主刻度的间距应随 `X` 改变而变化，使得主刻度在屏幕上的间距大致落在可读范围（实现可通过选择使 `majorStep` 最接近 `targetMajorPx * X` 的候选值达成，`targetMajorPx` 为实现固定常量）。
2. **副刻度（Minor）**
   - 当 `majorStep` 的首位因子为 `1` 或 `5` 时，副刻度步长 `minorStep = majorStep / 5`，因此相邻两主刻度之间应出现 4 个副刻度。
   - 当 `majorStep` 的首位因子为 `2` 时，副刻度步长 `minorStep = majorStep / 4`，因此相邻两主刻度之间应出现 3 个副刻度。
3. **标注**
   - 必须在主刻度位置绘制坐标数值（世界单位），副刻度不标注数值或使用同等清晰策略（但必须与现有视觉层级一致）。
   - 当主刻度位置与世界坐标对应时，坐标数值必须与世界坐标一致（允许实现使用合理的取整策略）。

#### 场景:缩放/平移后刻度更新

- **当** 用户在地图展示区域滚轮缩放或左键拖动平移
- **那么** 左侧与上侧的主/副刻度间距与标注的数值必须同步更新，且刻度应始终与当前可视区域的世界坐标对齐

#### 场景:主刻度首位为 1/5 时的副刻度数量

- **当** 当前计算得到的 `majorStep` 的首位因子为 `5`（即 `majorStep = 5 * 10^n`）
- **那么** 相邻两主刻度之间应出现 4 个副刻度，且副刻度数值步长为 `majorStep/5`
- **并且** 当可视区域跨越主刻度 `3000、3500、4000` 时，副刻度必须包含 `3100、3200...`（具体包含范围受可视窗口截断影响）

#### 场景:主刻度首位为 2 时的副刻度数量

- **当** 当前计算得到的 `majorStep` 的首位因子为 `2`（即 `majorStep = 2 * 10^n`）
- **那么** 相邻两主刻度之间应出现 3 个副刻度，且副刻度数值步长为 `majorStep/4`

### 需求:倍率展示与滚轮缩放

系统必须从当前地图的 `.json` 中读取 `scale` 数值。系统必须在左侧展示框左下角显示当前倍率文案，格式必须能够理解为「每屏幕像素对应的游戏单位数」，且对于边长为 \(S\) 像素的正方形展示框与 `scale` 值，展示所依据的倍率 \(X\) 必须满足 \(X = (1024 / S) \times \text{scale}\)（与用户在需求中给出的示例一致）。当用户鼠标悬浮在左侧地图区域上时，系统必须支持滚轮放大或缩小地图显示；缩放后 \(S\) 或等效显示尺寸与倍率 \(X\) 必须同步更新。地图显示缩放必须不得小于「整图刚好适配展示框」时的状态；必须不得大于「\(1\) 屏幕像素对应 \(1\) 游戏单位」即 \(X = 1\) 时的状态。

#### 场景:悬停时滚轮改变缩放

- **当** 鼠标指针位于左侧地图展示区域内且用户滚动鼠标滚轮
- **那么** 地图的显示缩放必须在上述最小与最大界限内变化，且左下角倍率文案必须反映当前 \(X\)，且缩放后视口内**不得**出现白边或未覆盖区域（见「地图铺满与白边」需求）

### 需求:地图右下角距离叠加展示

系统必须在左侧地图展示框**右下角**以**叠加层**形式展示当前用于测距与公式计算的**游戏单位距离**及米制换算。该叠加层文案的**字体、字号、颜色、内边距、对齐方式及背景/描边（若存在）**必须与同一展示框**左下角**「每屏幕像素对应游戏单位」倍率文案的实现对用户可见效果一致（允许通过复用同一样式类或 design token 达成）。展示格式必须为：`游戏单位距离的十进制表示` + 项目约定的游戏单位文案 + 空格 + `(` + 按 **1 游戏单位 = 25.4 mm** 换算的米制数值（实现应采用固定小数位数，例如保留两位小数）+ ` m)`。

#### 场景:右下角与左下角视觉一致

- **当** 用户已加载地图且地图展示框同时显示左下角倍率与右下角距离
- **那么** 两处文案的视觉层级与样式必须一致或刻意复用同一套样式，且右下角必须呈现「数值 + 单位 + ` (xx.xx m)`」形式

#### 场景:无圆心时右下角显示零距离

- **当** 用户尚未完成第一次有效测距落点，或已右键清除测距且当前无圆心
- **那么** 右下角距离叠加层仍必须可见，且游戏单位距离必须显示为 0，米制括号必须显示为 ` (0.00 m)`（或实现约定的等价零值格式化）

### 需求:左键拖动平移地图

系统必须支持用户在左侧地图展示区域内**按住鼠标左键并移动**（拖动）时**平移**地图图层（与滚轮缩放、测距圆叠加层同步变换）。平移量必须与用户拖动位移一致或按实现采用等效映射，且平移后必须仍满足「地图铺满与白边」需求中的不露空约束。

#### 场景:按住左键拖动

- **当** 用户在地图展示区域内按下左键并按住不放且移动鼠标
- **那么** 地图必须随拖动平移，且视口内不得露出白边或未覆盖区域

### 需求:地图铺满与白边

**无论**当前缩放比例、平移偏移或窗口尺寸如何，左侧正方形展示框内**必须**被地图像素完全覆盖，**禁止**出现白边、未绘制区域或页面背景从地图与视口之间露出。实现必须通过「最小缩放不低于铺满视口所需尺度」与「平移夹紧」等机制保证；若用户操作会导致露空，系统必须拒绝该操作或自动修正平移/缩放。

#### 场景:缩放与平移后不露白

- **当** 用户完成滚轮缩放或左键拖动平移（或二者组合）
- **那么** 展示框内必须仍无白边或未覆盖区域

### 需求:圆形测距与交互状态

系统必须通过**单击**左键（按下并释放，且移动距离未超过实现约定的**拖动阈值**，因而被判定为点击而非拖动）在地图上记录圆心位置。**在用户第一次有效单击并释放之后**，系统必须立即在以该点为圆心的位置绘制**红色**参考圆与**绿色**参考圆。绿色圆的半径（游戏单位）必须为 \(\text{baseDamage} \times 3.5\)。两参考圆须随地图缩放与倍率 \(X\) 正确换算为屏幕像素。

红圈目标伤害阈值必须由当前滑块值决定：当 \(HP \ge armor\) 时，目标伤害 \(T = HP + armor\)；当 \(HP < armor\) 时，目标伤害 \(T = HP \times 2\)。红色圆的半径（游戏单位）必须按当前伤害公式对目标伤害 \(T\) 反解到距离 \(d\)（\(\text{damage} = \text{baseDamage} \times e^{-\frac{1}{2}\left(\frac{6 d}{7 \cdot \text{baseDamage}}\right)^2}\)）。若目标伤害不可达（例如 \(T > \text{baseDamage}\) 或 \(T \le 0\)），系统必须不绘制红圈或采用明确的降级显示。**因距离大于 \(3.5 \times \text{baseDamage}\) 时伤害为 0，红圈半径（游戏单位）必须不得大于绿圈半径 \(3.5 \times \text{baseDamage}\)**；若反解得到的 \(d\) 大于该值，必须将红圈半径钳制为与绿圈相同。

在第一次有效单击**释放之后**、用户**未按住任何鼠标键**移动指针的过程中，系统必须绘制测距圆，其圆心必须为第一次单击位置，半径必须由圆心到当前指针位置的欧氏距离（像素）确定。系统必须计算**距离**（游戏单位）：该距离必须等于像素欧氏距离乘以当前倍率 \(X\)。**距离的屏幕展示位置必须满足**「地图右下角距离叠加展示」需求。用户第二次**单击**左键时，若当前存在未固定的测距圆，系统必须将该测距圆与圆心固定，且必须停止随指针更新距离与伤害数值；**参考红圈与绿圈**必须保持显示直至清除。用户按下鼠标右键时，系统必须清除测距圆、圆心、固定状态及**两条参考圆**，并必须允许重新开始测距。

#### 场景:红圈阈值随 HP 护甲变化

- **当** 用户调整 HP 或护甲滑块
- **那么** 红圈半径必须按新的目标伤害阈值 \(T\) 实时更新

#### 场景:右键清除

- **当** 用户在地图区域点击鼠标右键
- **那么** 测距圆、圆心、红/绿参考圆必须被清除，且用于公式与展示的测距距离必须恢复为 **0**（见「距离米制换算显示」与「元数据读取与伤害计算」），右侧信息区伤害相关展示必须与 \(d = 0\) 一致

### 需求:距离米制换算显示

地图右下角叠加层（见「地图右下角距离叠加展示」）必须在**同一行**展示游戏单位距离与米制距离。换算必须固定为：**1 游戏单位 = 25.4 mm**（米制长度 = 游戏单位距离 × \(25.4 / 1000\) m）。**无论**当前是否存在测距圆心，**只要**地图已处于可测距状态，该叠加层必须始终展示游戏单位数值与 ` (… m)` 括号内容；当无圆心或已清除时，游戏单位距离必须为 **0**，米制展示必须为对应的零值格式化（例如 `0.00 m`）。

#### 场景:括号内显示米

- **当** 地图已展示且右下角距离叠加层可见
- **那么** 该行必须呈现「数值 + 游戏单位文案 + ` (… m)`」形式，其中括号内为按 1 unit = 25.4 mm 换算的米制距离

#### 场景:无圆心时仍显示米括号且为零

- **当** 用户尚未落点测距或已清除测距
- **那么** 右下角距离行必须仍包含米制括号内容，且数值对应 **0** 游戏单位

### 需求:HP 与护甲滑块输入

系统必须在右侧信息区「计算伤害」下方提供两个滑块输入：初始 HP 与初始护甲值。两个滑块的最小值必须为 0，最大值必须为 100，步长必须为 1。

#### 场景:滑块范围与步长

- **当** 用户查看右侧参数区
- **那么** 必须看到「初始 HP」与「初始护甲」两个滑块，且二者均满足 0-100、步长 1

### 需求:爆炸后状态展示

系统必须在两个滑块下方展示一行爆炸后状态。该行必须显示：`HP:XX.XX (XX)，护甲：XX`。其中 HP 主值为浮点数，括号内必须为 HP 的向上取整整数，护甲值必须为整数。

#### 场景:格式化显示

- **当** 存在有效伤害并完成结算
- **那么** 系统必须按 `HP:浮点(向上取整整数)，护甲:整数` 的格式渲染结果

### 需求:元数据读取与伤害计算

系统必须从当前地图的 `.json` 中读取 `c4_base_damage` 作为基本伤害。**系统不得在右侧信息区单独占用一行展示当前距离**（距离由地图右下角叠加层展示，并须满足「距离米制换算显示」）。在地图选择框下方，**第二行**必须展示基本伤害，**第三行**展示计算伤害；其下为 HP/护甲滑块与爆炸后状态行。设 **距离** 为游戏单位 \(d\)（用于公式与右侧展示；**无测距圆心或已清除后** \(d\) 必须为 **0**），**基本伤害** 为 \(\text{baseDamage}\)。当 \(d > 3.5 \times \text{baseDamage}\) 时，第三行展示的**计算伤害必须**为 \(0\)。当 \(d \le 3.5 \times \text{baseDamage}\) 时，计算伤害必须按下式计算：\(\text{damage} = \text{baseDamage} \times e^{-\frac{1}{2}\left(\frac{6 \times d}{7 \times \text{baseDamage}}\right)^2}\)。

伤害结算到 HP/护甲时必须遵循：若初始护甲 \(> damage \times 0.5\)，则护甲减少 \(\lfloor damage \times 0.5 \rfloor\)，HP 减少 \(damage \times 0.5\)；否则护甲减少到 0，HP 减少 \(damage - \text{初始护甲}\)。当结算后 HP 小于 0 时，结果行必须使用红色字体显示 `HP:0(0)，护甲：0`；否则结果行必须使用黑色字体显示实际结算结果。

当圆被固定后，距离与伤害数值必须保持固定直至右键清除或更换地图。

#### 场景:显示基本伤害与公式伤害

- **当** 用户已选择地图且 `.json` 中存在有效的 `c4_base_damage`
- **那么** 第二行必须显示该基本伤害值，且第三行必须按上条规则显示计算伤害（含 \(d = 0\) 时按公式计算的结果）

#### 场景:进入页面未选点时按零距离计算

- **当** 用户已选择地图且元数据有效，但尚未在地图上进行第一次有效测距落点
- **那么** 公式所使用的 \(d\) 必须为 **0**，第三行计算伤害与爆炸后状态结算必须与 \(d = 0\) 一致

#### 场景:远距离伤害为零

- **当** 当前测距距离 \(d\) 严格大于 \(3.5 \times \text{baseDamage}\)
- **那么** 第三行计算伤害必须显示为 \(0\)

#### 场景:护甲充足分支

- **当** 初始护甲 \(> damage \times 0.5\)
- **那么** 系统必须按「护甲减 \(\lfloor damage \times 0.5 \rfloor\)、HP 减 \(damage \times 0.5\)」结算

#### 场景:护甲不足分支

- **当** 初始护甲 \(\le damage \times 0.5\)
- **那么** 系统必须把护甲结算为 0，且 HP 扣减值为 \(damage - \text{初始护甲}\)

#### 场景:HP 为负红色展示

- **当** 结算后 HP \(< 0\)
- **那么** 系统必须以红色显示固定文本 `HP:0(0)，护甲：0`
